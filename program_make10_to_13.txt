# 9. Target v1.3.0 Pipeline

The final v1.3.0 pipeline should be conceptually:

```txt
single-image mode:
input image
-> optional worker or main-thread processing
-> optional source preprocessing
-> tile conversion
-> palette source / palette mapping
-> dithering if enabled
-> palette editor edits if any
-> icon assist outline if enabled
-> final canvas
-> preview/export

layered mode:
multiple layer images
-> each layer processed independently using shared settings
-> each processed layer becomes a layer result canvas
-> ordered visibility-aware composite preview
-> PNG/JPG export uses flattened composite
-> Aseprite export preserves separate layers as separate layer/cel data
```

Do not pixelize the flattened layer composite first.

Layered mode must process layers independently first, then composite only for preview or flattened PNG/JPG export.

All export formats must use the final processed result, not a preview-only image.

# 10. v1.0.0 — Performance Stabilization / Web Worker

## Goal

Move expensive processing work off the UI thread where practical.

The current app has many main-thread operations:

* preprocessing
* tile conversion
* palette mapping
* dithering
* palette analysis
* outline
* export preparation

The v1.0.0 goal is to reduce UI freezes for large images while preserving the existing app behavior.

## Required Scope

Add a worker-backed processing path for expensive conversion work.

Recommended file structure:

* `src/conversionWorker.js`
* `src/workerClient.js` or equivalent main-thread wrapper

If new files are created, load them through normal script tags where needed.

Do not use ES Modules.

Do not introduce a build step.

## Worker Compatibility

Important:

The app should still run by opening `index.html` directly when possible, but Web Workers may be restricted in some `file://` browser contexts.

Therefore:

* worker path must be optional
* if worker initialization fails, fallback to existing main-thread conversion
* show a non-blocking warning or status text if fallback is used
* document that using `python -m http.server 8000` is recommended for full worker behavior

Do not break direct `index.html` execution.

## Worker Processing

The worker should handle, if practical:

* source preprocessing
* tile conversion
* palette mapping
* dithering
* palette analysis
* outline processing

If some steps are safer to keep on the main thread, document the limitation.

The worker should receive serializable data only.

Use transferable objects where practical:

* `ArrayBuffer`
* `ImageData.data.buffer` if safe

Do not attempt to send DOM nodes, canvas elements, or image elements directly to the worker.

## Progress and Cancel

Add processing status UI.

Required:

* show when processing is running
* show progress stage text
* disable conflicting controls during active processing where necessary
* add cancel support if practical

Recommended stage labels:

```txt
이미지 준비 중
전처리 중
픽셀 변환 중
팔레트 처리 중
디더링 처리 중
외곽선 처리 중
미리보기 갱신 중
```

If true percentage progress is hard, use stage-based progress.

Cancel behavior:

* user can cancel a long conversion
* canceled result must not overwrite the current valid result
* UI should return to a stable state
* show a Korean message such as `변환을 취소했습니다.`

If cancel is too risky, implement stage status only and document cancel as not included.

## Large Output Behavior

Keep existing large-output warning behavior.

Do not repeatedly auto-convert very large output sizes.

Web Worker should improve responsiveness, not remove validation/warnings.

## v1.0.0 Tests

Add or update tests for:

* default behavior unchanged
* worker path available when supported
* worker fallback works when worker creation fails
* conversion result from worker matches main-thread result for controlled images
* progress/status UI appears during processing
* cancel does not overwrite current result if implemented
* PNG/JPG/Aseprite export still use final canvas
* no ES Module import/export
* no browser alert
* no simple resize shortcut

Update all docs and changelog after this milestone.

# 13. v1.3.0 — Layered PNG Input / Layered Aseprite Export

## Goal

Add a layered workflow for users who have separated images and want each layer pixelized independently.

This is one of the strongest workflow features for Aseprite editing.

Do not implement PSD import.

Do not implement GIF animation.

Do not implement multi-frame animation.

Do not implement general batch conversion / ZIP export.

## Input Model

Layered mode accepts multiple local image files as layers.

Supported file types should follow existing image support:

* PNG
* JPG
* JPEG

Recommended:

* PNG is preferred because it preserves transparency.

Each file becomes one layer.

Layer data should include:

* id
* name
* source file name
* source width
* source height
* visibility
* order
* source image data
* processed layer canvas
* optional warning state

Do not store absolute local file paths.

## Layered Mode UI

Add a Layered Mode section or toggle.

Recommended Korean UI labels:

```txt
레이어 모드
레이어 추가
레이어 목록
이름 변경
표시 / 숨김
위로 이동
아래로 이동
삭제
합성 미리보기
레이어별 미리보기
```

Required UI features:

* add multiple image files as layers
* list layers
* rename layer
* reorder layers
* toggle visibility
* delete layer
* show composite preview
* allow selecting a layer to inspect its processed preview if practical

If the UI becomes crowded, use a collapsed Layered Mode section.

## Layer Size Policy

All layers should be processed to the same output size.

Use the existing global output size settings:

* Width/Height presets
* Custom size

Do not add per-layer output size in v1.3.0.

If source images have different dimensions:

Recommended safe behavior:

* use each layer's own source dimensions for tile sampling
* output every layer to the same resolved output width/height
* warn if source dimensions differ
* do not attempt complex positioning in v1.3.0

Alternative safe behavior if implementation requires it:

* create a shared source canvas based on the maximum source width/height
* draw each layer at top-left before per-layer conversion
* document that v1.3.0 does not support per-layer positioning

Do not silently distort layer alignment without documenting the policy.

## Layer Processing Pipeline

Process each layer independently.

Correct:

```txt
layer A source -> preprocess -> tile conversion -> palette/dither -> outline -> layer A result
layer B source -> preprocess -> tile conversion -> palette/dither -> outline -> layer B result
layer C source -> preprocess -> tile conversion -> palette/dither -> outline -> layer C result
then composite visible processed layers for preview and PNG/JPG export
```

Incorrect:

```txt
flatten all source layers -> pixelize once -> lose layer separation
```

Do not use the incorrect pipeline.

## Palette and Dithering Policy for Layers

Use shared global settings in v1.3.0.

Do not implement per-layer palette settings unless it is very safe.

Important:

* generated palette behavior must be clearly defined
* fixed built-in/imported palette behavior is straightforward and should map each layer to the same fixed palette
* generated palette can either:

  * generate palette per layer, or
  * generate one shared palette from composite/combined visible pixels

Choose the safer implementation and document it.

Recommended:

* for `builtIn` and `imported`, use the fixed shared palette across all layers
* for `generated`, use per-layer generated palette in v1.3.0 unless shared generation is simple and stable
* document the behavior clearly

## Preview and Export

Preview:

* composite visible processed layers in layer order
* hidden layers should not appear in composite preview
* selected layer preview shows that layer alone if implemented

PNG export:

* flattened composite of visible processed layers

JPG export:

* flattened composite of visible processed layers over white background

Aseprite export:

* preserve layers
* one frame
* one cel per visible or all layers, depending on safe exporter behavior
* preserve layer order
* preserve layer names
* use RGBA cel data
* hidden layers should either:

  * export as hidden layers if exporter supports visibility flags safely, or
  * be omitted from export and document this clearly

Recommended safe behavior:

* export visible layers as separate Aseprite layers
* omit hidden layers from Aseprite export in v1.3.0
* document hidden-layer export behavior clearly

Do not break existing single-layer Aseprite export.

## Aseprite Exporter Extension

Extend the existing Aseprite exporter carefully.

Current exporter likely creates:

* one frame
* one layer
* one raw RGBA cel

v1.3.0 should support:

* multiple layer chunks
* multiple cel chunks in one frame
* each cel uses the processed layer canvas
* each layer has a name
* dimensions match output width/height
* color depth remains 32-bit RGBA

Add tests that parse the exported binary enough to verify:

* valid Aseprite file magic
* valid frame magic
* output width
* output height
* 32-bit color depth
* expected number of layers
* expected number of cels
* layer names if parser support is practical

If layer name binary parsing is too much, at minimum verify layer/cel counts and dimensions.

## State Management

Layered mode must not destabilize single-image mode.

Recommended:

* keep existing single-image state intact
* add `layeredState` separately
* do not mix layer arrays into every single-image path unless needed

Possible state:

```txt
layeredState = {
  enabled,
  layers,
  selectedLayerId,
  compositeCanvas,
  lastProcessedLayerCanvases,
  dirty
}
```

When Layered Mode is off:

* existing single-image behavior should be unchanged

When Layered Mode is on:

* conversion uses layers
* preview uses composite
* export follows layered policies

## v1.3.0 Tests

Add or update tests for:

* layer rename works
