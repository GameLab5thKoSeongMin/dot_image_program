너는 지금부터 “32×32 도트 아이콘 생성기” 프로그램을 처음부터 끝까지 개발한다.

이 작업은 빠르게 끝내는 것이 목표가 아니다.

사용자는 Codex를 장시간 켜둘 예정이므로, 속도보다 정확성, 안정성, 구조적 완성도, 테스트, 문서화를 우선한다.

단, 불필요하게 복잡한 프레임워크나 과한 아키텍처를 도입하지 않는다.
프로그램 규모에 맞는 단순하고 견고한 구조를 유지한다.

---

# 핵심 목표

이 프로그램의 목표는 사용자가 PNG 또는 JPG 이미지를 입력하면, 입력 이미지를 32열 × 32행으로 분할하고, 각 타일의 대표색을 median 값으로 계산하여 최종 32×32 픽셀 아이콘 이미지를 생성하는 것이다.

중요하다.
이 작업은 단순히 이미지를 `32×32`로 resize하는 프로그램이 아니다.

반드시 원본 이미지를 32열 × 32행의 타일로 나누고, 각 타일 내부 픽셀들의 median 색상값을 구해서 결과 이미지의 픽셀 하나로 압축하는 방식으로 구현한다.

절대 단순히 다음 방식으로만 끝내지 마라.

```js
ctx.drawImage(image, 0, 0, 32, 32);
```

위 방식은 단순 리사이즈이므로 요구사항을 만족하지 않는다.

# 0-1. 실행 방식

이 프로젝트는 사용자가 최대한 쉽게 실행할 수 있어야 한다.

따라서 초기 버전에서는 ES Module 방식의 `import/export`를 사용하지 않는다.

즉, 다음과 같은 코드는 사용하지 않는다.

```js
export const OUTPUT_SIZE = 32;
import { OUTPUT_SIZE } from "./constants.js";
```

대신 `index.html`에서 일반 script 태그를 순서대로 로드한다.

```html
<script src="./src/constants.js"></script>
<script src="./src/fileHandler.js"></script>
<script src="./src/imageProcessor.js"></script>
<script src="./src/uiController.js"></script>
<script src="./src/app.js"></script>
```

각 파일은 전역 네임스페이스를 최소한으로 사용하되, 단순 실행 가능성을 우선한다.

예시:

```js
window.PixelIconConstants = {
  OUTPUT_SIZE: 32,
  TRANSPARENT_ALPHA_THRESHOLD: 16,
  MIN_OPAQUE_RATIO: 0.1,
  SUPPORTED_MIME_TYPES: ["image/png", "image/jpeg"],
  DEFAULT_OUTPUT_FILENAME: "pixel_icon_32x32.png"
};
```

최종 결과물은 가능하면 `index.html`을 브라우저에서 직접 열어 실행할 수 있어야 한다.

로컬 서버 실행이 필요하다고 판단될 경우, `README.md`에 반드시 실행 방법을 적는다.

예시:

```bash
python -m http.server 8000
```

```md
# AGENTS.md

## Project Goal
Build a local 32×32 pixel icon generator.

The app must convert an input PNG/JPG/JPEG image into a 32×32 pixel icon using tile-based median color sampling.

This must not be implemented as a simple image resize.

## Repository Layout
- `index.html`: Main HTML entry.
- `styles/style.css`: Layout and visual styling.
- `src/constants.js`: Global constants.
- `src/fileHandler.js`: File validation and filename handling.
- `src/imageProcessor.js`: Image loading, tile sampling, median color calculation, transparency handling.
- `src/uiController.js`: DOM updates and UI state.
- `src/app.js`: App entry point and event flow.
- `tests/test-cases.html`: Manual and visual test runner.
- `tests/testImageFactory.js`: Test image generation helpers.
- `README.md`: User-facing usage guide.
- `DESIGN_SPEC.md`: Feature and GUI design.
- `TEST_PLAN.md`: Test checklist and results.
- `DEVELOPMENT_REPORT.md`: Step-by-step progress report.
- `PLANS.md`: Current execution plan and step status.
- `CHANGELOG.md`: Version history.

## Engineering Rules
- Keep the app dependency-free unless absolutely necessary.
- Prefer Vanilla HTML/CSS/JavaScript.
- Do not use ES Module `import/export` in the initial version.
- The app should run by opening `index.html` directly in a browser when possible.
- Keep functions small and responsibility-focused.
- Do not mix image processing logic with DOM manipulation.
- Do not implement the converter as a simple `drawImage(image, 0, 0, 32, 32)` resize.
- Use tile-based median color sampling.
- Preserve transparent PNG behavior as much as possible.
- Use constants for output size and transparency thresholds.

## Required Verification
Before considering the task complete:
- Test PNG input.
- Test JPG/JPEG input.
- Test transparent PNG input.
- Test drag and drop.
- Test invalid file type.
- Test download.
- Confirm downloaded PNG is exactly 32×32.
- Confirm the algorithm uses tile median sampling, not simple resize.
- Confirm small images do not crash the converter.
- Confirm very large images generate a result.

## Documentation Rules
After each development step, update `DEVELOPMENT_REPORT.md`.

Keep `PLANS.md` updated as the current execution plan changes.

Documentation should be accurate and grounded in implemented behavior.

Do not mark untested behavior as verified.
```

## 5. `DESIGN_SPEC.md`

프로그램의 기능 설계와 GUI 설계를 정리하는 문서다.

```md
# Design Specification

## 1. 프로그램 목표
입력 이미지를 32×32 도트 아이콘으로 변환하는 프로그램이라는 목표를 설명한다.

## 2. 핵심 변환 방식
이미지를 단순 리사이즈하는 것이 아니라, 원본 이미지를 32열 × 32행 타일로 나누고 각 타일의 median 색상을 결과 픽셀로 사용한다고 명시한다.

- 입력 이미지를 32×32 타일로 분할
- 각 타일에서 RGBA median 계산
- 투명 픽셀 처리 기준
- 결과 이미지는 32×32 PNG

## 4. 투명도 처리 규칙
- alpha 값이 특정 기준보다 낮으면 투명 픽셀로 판단
- 타일 내 불투명 픽셀 비율이 낮으면 결과 픽셀을 투명 처리
- 기준값은 constants.js에서 관리

## 5. GUI 레이아웃
화면을 4분할 구조로 설계한다.

- 좌측 상단: 입력 이미지 미리보기
- 우측 상단: 결과 이미지 미리보기
- 좌측 하단: 이미지 입력 영역
- 우측 하단: 다운로드 및 파일 정보 영역

## 6. 주요 모듈 역할
- app.js
- imageProcessor.js
- fileHandler.js
- uiController.js
- constants.js

각 파일의 책임을 설명한다.

## 7. 확장 가능성
추후 추가할 수 있는 기능을 적는다.

예시:
- 16×16 / 64×64 출력 옵션
- average / median / dominant color 선택
- 팔레트 제한 기능
- 배경 제거 옵션
- dithering 옵션
- 확대 배율 조절
- 투명도 기준값 UI 조절
```

# 1. 최종 프로그램 목표

사용자는 프로그램에서 이미지를 업로드하거나 드래그 앤 드롭으로 입력할 수 있어야 한다.

입력된 이미지는 좌측 상단에 미리보기로 표시된다.

프로그램은 입력 이미지를 32×32 그리드로 나눈다.

각 그리드 셀, 즉 하나의 타일 안에 있는 픽셀들의 색상값을 분석하여 median RGB 또는 RGBA 값을 대표색으로 선택한다.

투명 배경 PNG의 경우 투명도를 보존해야 한다.

대표색으로 구성된 32×32 픽셀 이미지를 생성한다.

생성된 이미지는 우측 상단에 확대 미리보기로 표시한다.

우측 하단에는 생성된 파일 정보와 다운로드 버튼을 표시한다.

사용자는 생성된 32×32 PNG 파일을 다운로드할 수 있어야 한다.

# 3. 기능 요구사항

---

## 3-1. 이미지 입력

사용자는 PNG 또는 JPG 파일을 입력할 수 있어야 한다.

입력 방식은 다음을 지원한다.

1. 파일 선택 버튼
2. 드래그 앤 드롭

지원 확장자:

* `.png`
* `.jpg`
* `.jpeg`

지원 MIME 타입:

* `image/png`
* `image/jpeg`

지원하지 않는 파일을 입력하면 사용자에게 오류 메시지를 보여준다.

손상된 이미지이거나 브라우저가 로드할 수 없는 이미지라면 앱이 멈추지 않고 오류 메시지를 보여준다.

---

## 3-2. 32×32 타일링

입력 이미지를 32열 × 32행으로 나눈다.

즉, 총 1024개의 타일을 만든다.

원본 이미지의 가로 길이를 32등분하고, 세로 길이를 32등분한다.

각 타일은 최종 32×32 이미지의 픽셀 하나에 대응된다.

예시:

* 원본 이미지의 좌측 상단 타일 → 결과 이미지의 (0, 0) 픽셀
* 원본 이미지의 우측 하단 타일 → 결과 이미지의 (31, 31) 픽셀

타일 경계가 소수점으로 나뉘는 경우에도 전체 이미지 영역이 빠짐없이 포함되도록 처리한다.

---

## 3-3. 타일 경계 계산 규칙

32×32 변환 알고리즘은 다음 방식으로 타일 경계를 계산한다.

```js
const xStart = Math.floor(tileX * imageWidth / OUTPUT_SIZE);
const xEnd = Math.floor((tileX + 1) * imageWidth / OUTPUT_SIZE);
const yStart = Math.floor(tileY * imageHeight / OUTPUT_SIZE);
const yEnd = Math.floor((tileY + 1) * imageHeight / OUTPUT_SIZE);
```

단, 원본 이미지가 너무 작거나 타일 경계가 겹쳐서 빈 타일이 생기는 경우에는 최소 1픽셀 이상 샘플링하도록 보정한다.

예시:

```js
const safeXEnd = Math.max(xEnd, xStart + 1);
const safeYEnd = Math.max(yEnd, yStart + 1);
```

또한 이미지 범위를 넘지 않도록 clamp 처리한다.

```js
const clampedXStart = Math.max(0, Math.min(xStart, imageWidth - 1));
const clampedYStart = Math.max(0, Math.min(yStart, imageHeight - 1));
const clampedXEnd = Math.max(clampedXStart + 1, Math.min(safeXEnd, imageWidth));
const clampedYEnd = Math.max(clampedYStart + 1, Math.min(safeYEnd, imageHeight));
```

이 규칙을 통해 다음 이미지도 안정적으로 처리한다.

* 100×100
* 1920×1080
* 17×17
* 32보다 작은 이미지
* 가로세로 비율이 극단적인 이미지

---

## 3-4. 타일 대표색 계산

각 타일 내부의 픽셀을 수집한다.

각 채널별로 median 값을 구한다.

기본 계산 방식:

* R 채널 median
* G 채널 median
* B 채널 median
* A 채널 median

최종 픽셀 색상은 다음과 같이 한다.

```txt
(medianR, medianG, medianB, medianA)
```

단, PNG 투명 배경 처리를 위해 다음 규칙을 적용한다.

1. 타일 내 대부분의 픽셀이 투명하면 결과 픽셀도 투명하게 만든다.
2. alpha 값이 매우 낮은 픽셀은 색상 median 계산에서 제외할 수 있다.
3. 투명 판단 기준은 `constants.js`에서 상수로 관리한다.
4. 기본값은 `alpha < 16`이면 투명 픽셀로 판단한다.
5. 타일 내 불투명 픽셀 비율이 너무 낮으면 결과 픽셀을 완전 투명 처리한다.
6. 기본 불투명 픽셀 최소 비율은 `0.1`로 한다.
7. 투명 처리되지 않는 타일의 RGB median은 불투명 픽셀을 기준으로 계산한다.
8. 투명 처리되지 않는 타일의 alpha median도 불투명 픽셀을 기준으로 계산한다.
9. 타일 내 불투명 픽셀이 하나도 없으면 결과 픽셀은 `(0, 0, 0, 0)`으로 처리한다.

이 기준값은 나중에 조정할 수 있도록 변수로 분리한다.

---

## 3-5. 결과 이미지 생성

32×32 크기의 canvas 또는 이미지 데이터를 생성한다.

1024개 픽셀에 각 타일의 대표색을 적용한다.

결과 이미지는 PNG로 생성한다.

결과 미리보기는 실제 32×32 크기만 보여주면 너무 작으므로, 화면에서는 확대해서 보여준다.

확대 표시 시 픽셀아트 느낌이 유지되도록 CSS에서 다음 처리를 한다.

```css
image-rendering: pixelated;
image-rendering: crisp-edges;
```

---

## 3-6. 다운로드

생성된 결과 이미지를 PNG 파일로 다운로드할 수 있어야 한다.

파일명 기본값은 다음과 같이 한다.

```txt
pixel_icon_32x32.png
```

원본 파일명이 있을 경우 다음처럼 만든다.

```txt
원본파일명_32x32.png
```

다운로드는 canvas의 `toBlob` 또는 `toDataURL`을 사용해 구현한다.

결과 이미지가 없을 때 다운로드 버튼은 비활성화한다.

# 4. GUI 요구사항

GUI는 전체 화면을 4분할 구조로 설계한다.

단, 상단 영역이 하단 영역보다 더 크게 보여야 한다.

전체 레이아웃은 다음과 같다.

```txt
┌───────────────────────┬────────────────────────┐
│ 좌측 상단              │ 우측 상단               │
│ 입력 이미지 미리보기   │ 생성 이미지 미리보기     │
├───────────────────────┼────────────────────────┤
│ 좌측 하단              │ 우측 하단               │
│ 이미지 입력 영역       │ 파일 정보 / 다운로드     │
└───────────────────────┴────────────────────────┘
```

화면 비율은 대략 다음과 같이 한다.

* 상단: 전체 높이의 약 70%
* 하단: 전체 높이의 약 30%
* 좌측: 전체 너비의 50%
* 우측: 전체 너비의 50%

정확한 해상도는 자율적으로 선택해도 된다.

다만 최소한 다음 조건을 만족해야 한다.

1. 입력 이미지와 결과 이미지가 한 화면에서 비교 가능해야 한다.
2. 드래그 앤 드롭 영역이 명확해야 한다.
3. 다운로드 버튼이 명확해야 한다.
4. 오류 메시지가 사용자에게 보여야 한다.
5. 반응형 레이아웃을 어느 정도 고려한다.

# 5. GUI 세부 설계

---

## 5-1. 좌측 하단

좌측 하단에는 이미지 입력창을 표시한다.

* 파일 선택 버튼
* 지원 파일 형식 안내 문구
* 오류 메시지 표시 영역

사용자가 이미지를 드래그하면 시각적으로 hover 상태가 표시되어야 한다.

---

## 5-2. 좌측 상단

좌측 상단에는 입력된 원본 이미지를 표시한다.

이미지가 입력되기 전에는 placeholder를 보여준다.

예시 문구:

```txt
입력 이미지 미리보기
```

이미지가 너무 크더라도 영역 안에 맞게 표시한다.

## 5-3. 우측 상단

우측 상단에는 생성된 32×32 도트 이미지를 표시한다.

이미지가 생성되기 전에는 placeholder를 보여준다.

예시 문구:

```txt
생성된 32×32 아이콘 미리보기
```

결과 이미지는 확대해서 보여준다.

픽셀 경계가 흐려지지 않도록 pixelated 렌더링을 적용한다.

## 5-4. 우측 하단

우측 하단에는 생성된 이미지 파일 정보를 표시한다.

다음 요소를 포함한다.

# 7. 코드 품질 요구사항

코드는 읽기 쉽게 작성한다.

중요한 함수에는 주석을 작성한다.

각 파일은 자신의 책임을 벗어나지 않도록 한다.

특히 다음 함수는 분리해서 작성한다.

* 파일 유효성 검사 함수
* 이미지 로드 함수
* 32×32 변환 함수
* 타일 경계 계산 함수
* median 계산 함수
* 투명도 처리 함수
* 결과 미리보기 갱신 함수
* 다운로드 함수
* 오류 메시지 표시 함수

매직 넘버는 상수로 분리한다.

예시:

```js
OUTPUT_SIZE = 32
TRANSPARENT_ALPHA_THRESHOLD = 16
MIN_OPAQUE_RATIO = 0.1
```

`imageProcessor.js`에는 이미지 변환 알고리즘만 둔다.

`uiController.js`에는 DOM 조작과 화면 갱신만 둔다.

`fileHandler.js`에는 파일 검증과 파일명 생성 로직만 둔다.

`app.js`는 전체 흐름을 연결하는 진입점 역할만 담당한다.

# 8. 알고리즘 구현 주의사항

이 프로그램의 핵심은 32×32 변환 알고리즘이다.

반드시 다음 방식으로 처리한다.

1. 원본 이미지를 임시 canvas에 원본 크기로 그린다.
2. 원본 이미지의 픽셀 데이터를 가져온다.
3. 원본 이미지를 32열 × 32행으로 나눈다.
4. 각 타일의 픽셀들을 수집한다.
5. 각 타일마다 R, G, B, A 채널의 median 값을 계산한다.
6. 계산한 median 값을 결과 이미지의 해당 픽셀에 넣는다.
7. 결과 32×32 canvas를 생성한다.
8. 결과 canvas를 미리보기와 다운로드에 사용한다.

타일 크기가 정수로 딱 나누어떨어지지 않는 이미지도 처리할 수 있어야 한다.

예시:

* 원본 이미지가 100×100이어도 32×32로 처리되어야 한다.
* 원본 이미지가 1920×1080이어도 32×32로 처리되어야 한다.
* 원본 이미지가 17×17처럼 32보다 작아도 최대한 안정적으로 처리되어야 한다.

# 9. 테스트와 검증에 대한 추가 요구

완성했다고 판단하기 전에 반드시 다음을 확인한다.

1. 결과 canvas의 width가 정확히 32인지 확인한다.
2. 결과 canvas의 height가 정확히 32인지 확인한다.
3. 다운로드된 PNG가 실제 32×32인지 확인한다.
4. 코드에서 단순 `drawImage(image, 0, 0, 32, 32)`만으로 변환을 끝내지 않았는지 확인한다.
5. `imageData`를 직접 순회하며 타일별 median을 계산하는 코드가 존재하는지 확인한다.
6. 투명 픽셀 판단 기준이 상수화되어 있는지 확인한다.
7. 파일 입력 전 다운로드 버튼이 비활성화되어 있는지 확인한다.
8. 잘못된 파일 형식 입력 시 앱이 깨지지 않는지 확인한다.
9. 매우 작은 이미지 입력 시 빈 타일로 인해 오류가 발생하지 않는지 확인한다.
10. 매우 큰 이미지 입력 시 결과가 생성되는지 확인한다.
