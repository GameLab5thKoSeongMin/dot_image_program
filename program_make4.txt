# Required Change 1. Simplify Output Size Controls

The current preset UI may show combined sizes such as:

* `16×16`
* `24×24`
* `32×32`
* `48×48`
* `64×64`

Replace this with separate width and height controls.

New size control structure:

```txt
Width
[16] [32] [64] [Original] [Custom]

Height
[16] [32] [64] [Original] [Custom]
```

Default:

* Width: `32`
* Height: `32`

Remove or de-emphasize combined size presets such as `24×24` and `48×48`.

Users can still create `24×24`, `48×48`, or any other valid size through `Custom`.

The purpose is to make the UI cleaner and easier to understand.

# Required Change 2. Custom Inputs Hidden Until Needed

Custom width input must be hidden until the user selects `Custom` for Width.

Custom height input must be hidden until the user selects `Custom` for Height.

Examples:

When Width is `32`:

```txt
Width
[16] [32] [64] [Original] [Custom]
```

When Width is `Custom`:

```txt
Width
[16] [32] [64] [Original] [Custom]
Custom Width: [ 48 ]
```

Same behavior for Height.

Do not show custom numeric input fields all the time.

If existing custom width/height inputs already exist, reuse them if practical, but change their visibility behavior.

Do not delete working validation logic unless it must be adjusted for the new policy.

# Required Change 3. Add Original Size Option

Add `Original` as an option for both Width and Height.

Behavior:

* Width `Original` means `outputWidth = originalImageWidth`
* Height `Original` means `outputHeight = originalImageHeight`

Examples:

If the source image is `512×384`:

* Width Original → `512`
* Height Original → `384`

This allows the app to act as a palette reduction / color simplification tool while preserving source resolution.

This is useful when the user wants to apply palette limiting without reducing the image resolution.

## Original option before image load

Before a valid image is loaded:

* `Original` width option must be disabled, or
* selecting it must show a warning and not break the app

Same for Height `Original`.

Recommended behavior:

* disable `Original` options until a valid image is loaded
* show source image size after image load
* then enable `Original`

Example Korean warning text if needed:

```txt
이미지를 먼저 업로드해야 원본 크기를 선택할 수 있습니다.
```

# Required Change 5. Remove Hard 256×256 Output Limit

Remove the hard validation rule that blocks output width or height above `256`.

The new hard validation rules are:

* width must be an integer
* height must be an integer
* width must be `>= 1`
* height must be `>= 1`
* width must be `<= originalImageWidth`
* height must be `<= originalImageHeight`

Do not block output sizes merely because they exceed 256.

However, large output sizes may be slow, so show performance warnings instead.

---

# Required Change 6. Performance Warnings for Large Output Sizes

Use the existing warning banner UI for performance warnings.

Do not use browser `alert()`.

Suggested thresholds:

```txt
outputWidth * outputHeight > 65,536
→ show a moderate performance warning

outputWidth * outputHeight > 262,144
→ show a stronger performance warning
```

Example Korean warning texts:

```txt
출력 해상도가 커서 변환 시간이 길어질 수 있습니다.
매우 큰 출력 해상도입니다. 변환 중 브라우저가 잠시 멈춘 것처럼 보일 수 있습니다.
```

These warnings should not block conversion.

Only invalid values should block conversion.

Invalid examples:

* width <= 0
* height <= 0
* width is not an integer
* height is not an integer
* width > originalImageWidth
* height > originalImageHeight

# Required Change 7. Avoid Repeated Heavy Auto-Conversions

If the current app automatically reconverts on every option change, review whether this causes performance problems for large output sizes.

Recommended behavior:

* Small sizes may auto-update if already implemented and stable.
* For large output sizes, avoid repeated automatic reconversion.
* Prefer an explicit `Apply` / `Convert` button behavior if needed.
* If an explicit apply button already exists, reuse it.
* If no apply button exists and auto-conversion is used, introduce debounce or block repeated heavy conversions for large outputs.

If you add or improve an explicit button, use Korean UI text such as:

```txt
적용
변환하기
미리보기 갱신
```

Do not overcomplicate the flow.

The key goal is to avoid UI lag when users select large output sizes such as Original width/height.

Fix this.

Recommended placeholder UI text in Korean:
