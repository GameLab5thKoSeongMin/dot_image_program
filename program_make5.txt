# Required Change 6. Keep Sampling Modes and Add One New Mode

Keep the existing sampling modes:

* `median`
* `average`
* `center`

Add one new sampling mode:

* `dominant`

The new `dominant` mode should choose the dominant visible color inside each source tile.

Purpose:

* reduce unwanted blended colors
* improve pixel-art-like color stability
* help palette limit produce cleaner results
* make Aseprite cleanup easier

## Dominant sampling behavior

Implement a practical browser-friendly dominant color method.

Recommended approach:

1. Collect visible pixels in the tile.
2. Exclude pixels whose alpha is below the existing transparency threshold.
3. Quantize RGB into buckets to avoid every tiny variation becoming a separate color.
4. Count bucket frequency.
5. Choose the most frequent bucket.
6. Use the average RGB of pixels inside that bucket as the tile output color.
7. Handle alpha carefully:

   * either use the average alpha of the winning bucket, or
   * snap visible alpha to 255 if alpha cleanup mode is active
8. If there are no visible pixels, return transparent.

Do not implement a heavy iterative k-means method for per-tile sampling in this task.

The goal is stable dominant color sampling, not expensive clustering.

Recommended helper functions:

* `calculateTileDominantColor(...)`
* `getDominantColorBucket(...)`
* `quantizeColorForDominantBucket(...)`

Add `dominant` to UI sampling options.

Example Korean UI text:

```txt
Dominant
```

or

```txt
대표색
```

Keep code identifier as `dominant`.

# Required Change 7. Fix Palette Alpha Bug

There is a known palette limit bug:

When pixels have many different alpha values, the palette color reduction does not effectively remove colors.

This must be fixed.

The current implementation may reduce visible RGB color count while preserving many alpha values. That can make the final output still contain many unique RGBA colors.

For pixel-art editing, the user wants the palette-limited result to actually reduce editable color complexity.

## Required alpha policy update

Add alpha cleanup / alpha quantization behavior for palette limiting.

Recommended default when palette limit is enabled:

* pixels with alpha below `TRANSPARENT_ALPHA_THRESHOLD` become fully transparent with alpha `0`
* visible pixels become fully opaque with alpha `255`

This is the simplest pixel-art-friendly behavior.

If preserving semi-transparent alpha is still desired, add an option such as:

* `alphaMode: "binary"` default for palette-limited output
* `alphaMode: "preserve"` optional

But do not overcomplicate the UI unless necessary.

Minimum acceptable implementation:

* When palette limit is enabled, normalize alpha so that the output does not contain many unique alpha values.
* Transparent pixels stay transparent.
* Visible pixels should not keep hundreds of distinct alpha values.
* The visible color count / unique RGBA count after palette limit should be meaningfully reduced.

Recommended behavior:

```txt
palette off:
- preserve existing alpha behavior

palette on:
- alpha < threshold -> alpha 0
- alpha >= threshold -> alpha 255
- map RGB to limited palette
```

This may remove anti-aliased semi-transparent edges, but that is acceptable for pixel-art-oriented editing.

Document this clearly.

## Counting rule update

Update color count display and tests to distinguish:

* visible RGB color count
* unique RGBA color count

The bug fix should ensure that when palette limit is enabled:

* visible RGB color count <= effective palette count
* unique RGBA color count should also be controlled and should not explode due to many alpha values

If binary alpha is used, unique RGBA count should generally be <= effective palette count + transparent color.
