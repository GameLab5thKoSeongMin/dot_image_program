# 12. v0.6.0 — Dithering

## Goal

Add dithering support to improve visual quality after palette limitation or fixed-palette mapping.

Dithering should help gradients and photos look better when color count is reduced.

## Required Dithering Modes

Add a Dithering option with:

* `off`
* `floydSteinberg`
* `bayer4x4`

Optional only if safe:

* `atkinson`
* `bayer2x2`

Default:

* `off`

Do not make dithering enabled by default.

## Dithering Behavior

Dithering should apply only when palette mapping is active.

Palette mapping is active when:

* palette mode is not `off`, or
* imported/fixed palette mapping is active after v0.7.0

If palette is off and no fixed palette is selected, dithering should either:

* be disabled, or
* show a warning that dithering requires a palette

Recommended Korean warning:

```txt
Dithering은 palette 제한 또는 외부 palette가 적용된 상태에서 사용할 수 있습니다.
```

## Floyd-Steinberg

Implement Floyd-Steinberg error diffusion for RGB values.

Rules:

* operate on visible pixels only
* preserve transparent pixels
* apply existing palette alpha behavior
* distribute error to neighboring pixels
* do not diffuse into fully transparent pixels if it causes visible artifacts
* keep alpha behavior consistent with palette mode

## Bayer 4x4

Implement ordered Bayer 4x4 dithering.

Rules:

* use a standard 4x4 Bayer threshold matrix
* apply before nearest-palette mapping or during palette mapping
* keep behavior deterministic
* preserve transparent pixels

## Dithering Strength

Add a simple dithering strength control only if practical.

Recommended:

* range: `0` to `1`
* default: `1`
* UI label may be Korean: `디더링 강도`

If this adds too much UI complexity, use fixed strength and document it.

## Suggested File Structure

You may implement dithering in:

* `src/paletteQuantizer.js`

or create:

* `src/ditheringProcessor.js`

If a new file is created, load it with normal script tags.

Do not use ES Modules.

## v0.6.0 Tests

Add tests for:

* dithering `off` preserves previous behavior
* Floyd-Steinberg works with palette `16`
* Bayer 4x4 works with palette `16`
* dithering preserves transparent pixels
* dithering does not run when palette is off unless intentionally supported
* PNG export uses dithered result
* JPG export uses dithered result composited on white
* Aseprite export uses dithered result
* default behavior remains unchanged

Update docs and changelog after this milestone.

# 13. v0.7.0 — External Palette / Palette Presets / Palette Source

## Goal

Add palette source selection so users can reduce images to a chosen palette, not only an automatically generated median-cut palette.

This is important for consistent game icon style.

## Palette Source Modes

Add a Palette Source option:

* `generated`
* `builtIn`
* `imported`

Default:

* `generated`

Existing palette behavior should map to `generated`.

## Generated Palette

This is the current behavior:

* median cut quantization
* auto / numeric / custom palette count

Keep it.

## Built-in Palettes

Add a small set of safe built-in palettes.

Recommended built-ins:

* `grayscale4`
* `gameboyLike4`
* `picoLike16`
* `warm8`
* `cool8`

Important:

* Do not falsely label a palette as an official copyrighted or exact third-party palette unless exact values are verified in the repository.
* If exact third-party palette values are not already known and verified in the repository, avoid claiming exact official palettes.
* It is acceptable to use clearly named generic palettes such as `gameboyLike4`, `warm8`, `cool8`.

Built-in palette entries should have:

* id
* display name
* hex color list

## Imported Palette

Add support for importing or pasting palettes.

Required import methods:

1. Paste HEX list into a textarea.
2. Parse `.txt` or `.hex` file containing HEX colors.

Optional only if safe:

3. Parse `.gpl` GIMP palette files.

Do not implement URL fetching in this task.

Avoid CORS, network, and security complexity.

## HEX Parsing Rules

Support:

```txt
#000000
#ffffff
000000
ffffff
#ff0000, #00ff00, #0000ff
```

Validation:

* accept 3-digit HEX if easy, normalize to 6-digit
* accept 6-digit HEX
* reject invalid colors
* remove duplicates
* preserve order where possible
* minimum colors: `2`
* maximum imported colors: `256`

Use warning banner for invalid input.

Example Korean warning:

```txt
유효한 HEX 색상이 필요합니다.
palette 색상은 2개 이상이어야 합니다.
palette 색상은 최대 256개까지 지원합니다.
```

## Fixed Palette Mapping

When palette source is `builtIn` or `imported`:

* map visible pixels to nearest palette color
* do not generate median cut palette
* palette count comes from the selected/imported palette
* preserve transparent pixels
* apply alpha normalization when palette mapping is active

For simplicity:

* keep existing palette alpha normalization when palette mapping is active
* document behavior clearly

## v0.7.0 UI

Add UI section:

```txt
Palette Source
[Generated] [Built-in] [Imported]
```

For Built-in:

* select dropdown for palette preset

For Imported:

* textarea for HEX input
* file input for `.txt`, `.hex`, optionally `.gpl`
* import/apply button
* imported palette preview

## v0.7.0 Tests

Add tests for:

* generated palette behavior still works
* built-in palette mapping works
* imported HEX list works
* duplicate HEX colors are removed
* invalid HEX shows warning
* imported palette with fewer than 2 colors is rejected
* imported palette above 256 colors is rejected
* transparent pixels remain transparent
* PNG/JPG/Aseprite export use fixed palette result
* default behavior remains unchanged

Update docs and changelog after this milestone.

# 14. v0.8.0 — Palette Swatch / Palette Editor / Color Merge

## Goal

Add palette swatch and simple editing tools so the user can inspect and clean the result before exporting to Aseprite.

This improves real icon editing workflow.

## Palette Swatch Display

Display the current result palette.

The palette swatch should show:

* color chip
* HEX value
* usage count if practical
* percentage usage if practical

The swatch should ignore fully transparent pixels for visible RGB count.

Show transparent count separately if practical.

Example Korean UI labels:

```txt
결과 팔레트
사용 색상 수
투명 픽셀 수
```

## Required Swatch Actions

Implement these actions:

1. Copy HEX color.
2. Replace color.
3. Merge one color into another color.

Optional only if safe:

4. Lock color.
5. Remove rarely used colors.
6. Sort by usage / hue / brightness.

## Replace Color

User selects a swatch color and enters or picks a replacement HEX.

Behavior:

* all pixels matching selected visible RGB color are replaced with new RGB
* preserve alpha according to current final canvas
* update preview
* update palette summary
* export uses edited result

## Merge Color

User selects source color and target color.

Behavior:

* all source color pixels become target color
* update preview
* update palette summary
* export uses edited result

This is useful for reducing similar colors before Aseprite export.

## Edited Result State

Important:

* palette edits must apply to the final processed canvas
* export must use edited canvas
* if user changes conversion settings, edits may be reset
* if edits are reset, show clear UI or document it

Recommended behavior:

* any new conversion clears manual palette edits
* show a small notice in Korean:

```txt
설정을 변경하면 수동 palette 편집 결과가 초기화됩니다.
```

## v0.8.0 Tests

Add tests for:

* swatch shows result colors
* HEX copy action exists
* replace color updates preview data
* merge color reduces visible color count
* export uses edited result
* changing conversion settings resets edits or handles them clearly
* transparent pixels are not incorrectly replaced as opaque colors
* default behavior remains unchanged

Update docs and changelog after this milestone.

## Padding / Centering
