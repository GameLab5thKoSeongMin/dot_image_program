# Main Goal of This Extension

Add a palette color limit feature to improve pixel-art consistency and make the generated icon easier to edit in Aseprite later.

This must be implemented as an extension of the current conversion pipeline.

The palette limit stage must be applied after tile-based conversion and before preview/export.

Pipeline concept:

input image
→ tile conversion using current sampling mode (`median` / `average` / `center`)
→ palette limit post-processing
→ preview
→ PNG / JPG / `.aseprite` export

Do not replace the current conversion logic.
Do not simplify the app into a different kind of tool.

Add palette color limiting with the following capabilities:

## 2. Auto Recommended Palette Count

Add an `auto` palette mode that chooses a recommended palette size based on output resolution.

Use these recommended defaults:

## 3. Palette Quantization Algorithm

Implement palette reduction using a real color quantization algorithm.

For this extension, use:

* `medianCut`

This should be the only required palette quantization algorithm for now.

Do not add K-means, Octree, external palette import, or dithering in this extension.

The palette limit stage must reduce the colors of the already-converted output image.

Recommended flow:

1. get result canvas or result `ImageData`
2. extract visible pixels only
3. build a limited palette using median cut
4. map visible pixels to the nearest palette color
5. preserve transparent pixels
6. create a new palette-limited result image or canvas

# Important Color Count Definition

For this extension, palette count means visible RGB palette count.

Do not count fully transparent pixels as palette colors.

Do not include pixels with alpha below the existing transparency threshold in palette generation.

For semi-transparent visible pixels:

* map their RGB value to the nearest palette color
* preserve their original alpha as much as possible

This means the output may still contain multiple alpha values, but the visible RGB colors should be limited to the selected palette count.

This is intentional and should be documented.

---

# Transparency Rules

Transparent pixels must not be included in palette generation.

Transparent pixels must remain transparent after palette quantization.

Rules:

* pixels with alpha below the existing transparency threshold are excluded from palette generation
* RGB quantization applies only to valid visible pixels
* alpha should be preserved as much as possible
* if a pixel is transparent, do not remap it to an opaque palette color
* if the entire image is transparent, return a transparent result without error
* if visible pixel count is `0`, do not attempt median cut; return the original transparent image data or an equivalent transparent image data

This is especially important for:

* PNG export
* JPG export preparation
* `.aseprite` export

For JPG export:

* first generate the palette-limited RGBA result
* then composite transparency onto white for JPG output
* the existing JPG transparency warning must remain correct
* because JPG does not support transparency, the final JPG may contain the white background color in addition to the palette-limited visible colors
* document this limitation clearly

---

# Palette Processing Edge Cases

Handle these cases safely:

1. If palette mode is `off`, do not alter the current conversion result.
2. If effective palette count is greater than or equal to the number of unique visible RGB colors, return the original result unchanged.
3. If the result has no visible pixels, return a transparent result unchanged.
4. If requested palette count is invalid, block the operation and show warning.
5. If requested palette count is `2`, still produce a valid result.
6. If requested palette count is `256`, still produce a valid result.
7. Median cut must not crash on tiny images, one-color images, transparent images, or images with fewer colors than the selected limit.
