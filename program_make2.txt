# New Functional Requirements

## 1. Variable Output Size

The app must no longer be restricted to only `32×32`.

Support the following preset sizes:

* 16×16
* 24×24
* 32×32
* 48×48
* 64×64

Also support custom output width and height input.

The output size must be defined by:

* `outputWidth`
* `outputHeight`

Do not keep the conversion logic hard-coded to a single `OUTPUT_SIZE` value.

Refactor the image conversion flow so that it supports arbitrary `outputWidth` and `outputHeight`.

However, preserve the default conversion settings as:

* `outputWidth = 32`
* `outputHeight = 32`
* `samplingMode = "median"`
* `outputFormat = "png"`

That means when the user opens the app and does nothing special, the app should still behave like the original 32×32 median PNG version.

---

## 2. Output Size Validation

Validate output size before conversion.

### Required validation rules

* Width and height must be integers.
* Width and height must be greater than or equal to 1.
* Width and height must not exceed the original image width and height.
* Width and height must not exceed `256`.
* The effective maximum for width is `min(originalImageWidth, 256)`.
* The effective maximum for height is `min(originalImageHeight, 256)`.

Examples:

If the input image is `300×200`:

* Allowed: `32×32`, `64×64`, `128×64`, `256×200`, `48×32`
* Rejected: `300×201`, `257×100`, `400×200`, `512×512`

If the input image is `20×20`:

* Allowed: `16×16`, `20×20`, `8×12`
* Rejected: `24×24`, `32×32`, `21×20`

If the input image is smaller than a preset size, that preset must either:

* be disabled, or
* produce a validation warning and block conversion

Choose the clearer UX, but validation must be obvious and conversion must not proceed with invalid output size.

Add a dedicated validation helper such as:

* `validateOutputSize(outputWidth, outputHeight, imageWidth, imageHeight)`

---

## 3. Sampling Mode Expansion

The current app uses median sampling.

Preserve that mode and add the following additional sampling modes:

* `median`
* `average`
* `center`

### Mode descriptions

#### median

Current existing behavior.

For each tile, compute the representative color using the median of each RGBA channel.

#### average

For each tile, compute the representative color using the arithmetic mean of the RGBA channels.

Use opaque pixels according to the same transparency handling rules used by the median mode.

#### center

For each tile, use the tile’s center pixel as the representative color.

If the exact center is fractional, choose a stable nearest pixel inside the tile.

Add a general dispatcher such as:

* `calculateTileRepresentativeColor(imageData, tileBounds, samplingMode)`

Recommended supporting functions:

* `calculateTileMedianColor(...)`
* `calculateTileAverageColor(...)`
* `calculateTileCenterColor(...)`

Do not remove or degrade the existing median implementation.

## 5. Output Format Selection

Add an output format selector.

The user must be able to choose one of the following output formats:

* `png`
* `jpg`
* `aseprite`

Default output format:

* `png`

### PNG export

PNG export must preserve transparency.

Use the existing canvas export flow if it is already working.

Recommended MIME type:

* `image/png`

### JPG export

JPG export must produce a valid `.jpg` or `.jpeg` file.

JPG does not support transparency.

Therefore, when JPG is selected:

* composite transparent pixels over a background color before export
* default JPG background color should be white
* show a warning if the image contains transparent pixels and JPG is selected
* do not silently preserve alpha because JPG cannot preserve alpha

Recommended warning text in Korean UI:

* `JPG는 투명도를 지원하지 않아 흰 배경으로 합성됩니다.`

Recommended MIME type:

* `image/jpeg`

Recommended quality:

* `0.92`

If adding a background color option is simple, add it.
If it would complicate the extension too much, keep the default white background and document the limitation.

### Aseprite export

Add `.aseprite` export support.

This must be a real `.aseprite` file writer, not a PNG file renamed with a `.aseprite` extension.

The `.aseprite` file should be generated from the converted output image, not from the original image.

Minimum target:

* single frame
* single normal image layer
* RGBA color mode
* canvas width = `outputWidth`
* canvas height = `outputHeight`
* pixel data based on the converted result canvas
* preserve alpha where possible
* file extension: `.aseprite`

Implement this in a focused module if needed, such as:

* `src/asepriteExporter.js`

If a new script file is added, load it in `index.html` before `app.js`.

Example script order:

```html
<script src="./src/constants.js"></script>
<script src="./src/fileHandler.js"></script>
<script src="./src/imageProcessor.js"></script>
<script src="./src/asepriteExporter.js"></script>
<script src="./src/uiController.js"></script>
<script src="./src/app.js"></script>
```

The Aseprite exporter should use a binary writer based on `ArrayBuffer`, `DataView`, or byte arrays.

Do not use external libraries unless absolutely necessary.

If external libraries are considered, document the reason first in `PLANS.md`, but prefer no dependency.

The exporter should follow the official Aseprite `.ase/.aseprite` file format specification.

The implementation must not pretend to support `.aseprite` if it only downloads a renamed PNG.

If full Aseprite compatibility cannot be verified in the environment:

* still implement a spec-based minimal writer
* add structural validation tests for the generated binary
* clearly document that external validation in the Aseprite desktop app is still recommended
* do not falsely claim that manual Aseprite desktop validation was completed

If the Aseprite executable or CLI is available in the environment, use it to validate that the exported `.aseprite` file opens correctly.

If Aseprite is not available, add a manual validation note in:

* `README.md`
* `USER_GUIDE_KO.md`
* `TEST_SUMMARY_KO.md`

### Aseprite export validation

Add a structural validation helper if practical, such as:

* `validateAsepriteBinary(arrayBuffer, expectedWidth, expectedHeight)`

Validation should check at least:

* file magic number
* frame count
* width
* height
* color depth is RGBA / 32 bpp
* at least one layer chunk exists
* at least one cel chunk exists
* generated file is not empty
* generated file extension is `.aseprite`

# Technical Migration Requirements

## 1. Refactor the conversion API carefully

The conversion entry point should evolve from a fixed-size structure into an option-based structure.

Recommended direction:

```js
convertImageToPixelIcon(image, options)
```

Where `options` includes:

* `outputWidth`
* `outputHeight`
* `samplingMode`

Also add a helper such as:

* `normalizeConversionOptions(options, imageWidth, imageHeight)`

Do not make unsafe breaking changes without checking how the current UI calls the converter.

If necessary, keep backward compatibility in the internal API while updating the main app flow.

---

## 2. Generalize tile boundary calculation

Current logic is based on 32×32 output.

Generalize it for arbitrary output width and height.

Tile bounds should use:

* input image width / height
* output width / height

Recommended logic:

```js
const xStart = Math.floor(tileX * imageWidth / outputWidth);
const xEnd = Math.floor((tileX + 1) * imageWidth / outputWidth);
const yStart = Math.floor(tileY * imageHeight / outputHeight);
const yEnd = Math.floor((tileY + 1) * imageHeight / outputHeight);
```

If bounds collapse, ensure at least one pixel is sampled.

Clamp bounds safely within the source image.

Preserve stability for:

* small images
* large images
* wide images
* tall images
* images smaller than default preset sizes
